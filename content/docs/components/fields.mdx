A flexible fields component system that displays multiple data fields in responsive layouts. Supports both read-only display and editable form inputs, with optional drag-and-drop reordering and dynamic field management.

## Props

### Fields Props

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `fields` | `FieldItemData[]` | - | Array of field objects to display |
| `layout` | `"grid" \| "stack" \| "inline"` | `"grid"` | Layout strategy for fields |
| `columns` | `1 \| 2 \| 3 \| 4 \| "auto"` | `2` | Number of columns for grid layout |
| `gap` | `"none" \| "sm" \| "md" \| "lg" \| "xl"` | `"md"` | Spacing between fields |
| `variant` | `"editable" \| "readonly" \| "compact"` | `"editable"` | Display mode for all fields |
| `size` | `"default" \| "sm" \| "lg"` | `"default"` | Size variant for all fields |
| `orientation` | `"vertical" \| "horizontal"` | `"vertical"` | Label/input orientation |
| `enableReordering` | `boolean` | `false` | Enable drag-and-drop reordering |
| `enableAddRemove` | `boolean` | `false` | Enable add/remove field buttons |
| `showSeparators` | `boolean` | `false` | Show separators between fields |
| `onChange` | `(fields) => void` | - | Called when fields array changes |
| `onFieldChange` | `(id, value) => void` | - | Called when a field value changes |
| `onFieldsReorder` | `(fields) => void` | - | Called after fields are reordered |
| `onFieldAdd` | `() => FieldItemData` | - | Function that returns new field data |
| `onFieldRemove` | `(id) => void` | - | Called when a field is removed |
| `addFieldLabel` | `string` | `"Add Field"` | Label for add field button |
| `showToolbar` | `boolean` | `enableAddRemove` | Show the toolbar |
| `toolbarActions` | `ReactNode` | - | Custom toolbar actions |
| `className` | `string` | - | Additional CSS classes |
| `classNames` | `object` | - | CSS classes for sub-components |

### FieldItemData Interface

| Property | Type | Description |
|----------|------|-------------|
| `id` | `string \| number` | Unique identifier for the field |
| `label` | `string` | Field label text |
| `value` | `string \| number` | Field value |
| `description` | `string` | Helper text below the field |
| `placeholder` | `string` | Placeholder text for inputs |
| `type` | `"text" \| "number" \| "email" \| "tel" \| "url" \| "date" \| "textarea" \| "select"` | Input type |
| `required` | `boolean` | Mark field as required |
| `disabled` | `boolean` | Disable field interaction |
| `error` | `string` | Error message to display |
| `options` | `Array<{label: string, value: string}>` | Options for select type |
| `rows` | `number` | Number of rows for textarea |
| `render` | `(field) => ReactNode` | Custom render function |

## Usage Guidelines

### When to Use Each Layout

**Grid**: Most common use case for dashboards, forms, and data displays. Automatically responsive with configurable columns.

**Stack**: Single-column layouts, mobile-first designs, narrow containers, or when each field needs maximum width.

**Inline**: Compact horizontal layouts for filters, toolbar controls, or dashboard metrics that should flow together.

### Variant Selection

**Editable**: Use for forms where users need to input or modify data. Includes full form controls with validation support.

**Readonly**: Use for displaying data that shouldn't be modified. Perfect for profile views, invoice details, or report summaries.

**Compact**: Use when space is limited and you need to fit more fields in a smaller area. Reduces padding and font sizes.

### Column Configuration

- **1 column**: Full-width fields, mobile layouts, forms with complex inputs
- **2 columns**: Standard forms, profile editors, settings panels (recommended default)
- **3 columns**: Dashboards, data grids, metric displays
- **4 columns**: Dense layouts, overview panels, admin interfaces
- **auto**: Responsive grid that adapts from 1→2→3→4 columns based on screen size

### Accessibility

- All fields include proper ARIA attributes (`aria-invalid`, `aria-describedby`, `aria-required`)
- Semantic HTML with `<label>` elements properly associated with inputs
- Keyboard navigation fully supported (Tab to navigate, Enter/Space for actions)
- Error messages announced to screen readers via `role="alert"`
- Required fields marked with visual and semantic indicators
- Drag handles keyboard accessible when reordering is enabled

### Best Practices

1. **Consistent IDs**: Always provide unique `id` values for fields to enable proper state management
2. **Meaningful Labels**: Use clear, concise labels that describe the expected input
3. **Helpful Descriptions**: Add descriptions for fields that need clarification or formatting guidance
4. **Validation Feedback**: Show errors inline with specific, actionable messages
5. **Responsive Design**: Test layouts at different breakpoints to ensure fields remain usable
6. **Field Types**: Use appropriate input types for better mobile keyboards and validation

## Field Types Reference

- **text**: Standard text input (default)
- **number**: Numeric input with increment/decrement controls
- **email**: Email input with validation and email keyboard on mobile
- **tel**: Telephone input with tel keyboard on mobile
- **url**: URL input with validation
- **date**: Date picker input
- **textarea**: Multi-line text input with configurable rows
- **select**: Dropdown selection with predefined options
